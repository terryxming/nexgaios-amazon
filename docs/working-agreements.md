# 工作约定（Terry 对本项目的偏好）

> 用户对"怎么干"给过的明确要求。工程通用纪律见仓库根 [../CLAUDE.md](../CLAUDE.md)；本文只记本项目特有的约定。


## 外行也要能看懂（硬要求）

SQP/仪表盘/分析的交付物和解读，必须**从"非亚马逊运营"视角衡量能不能看懂**——数据看不看得懂、术语懂不懂。看不懂就就地翻译成大白话。

**为什么**：报告读者不只有运营，还有老板/高管/外行；一份看不懂的报告 = 白做。Terry 本人也被术语卡过（问过 CRO 是什么）。

**怎么落地**：

- 交付物做**两层**：顶层白话可扫读（结论+关键数字），术语/推导/口径沉到"▸ 展开审计"层（这套"审计层"基调已在 Q1 切片定稿）。
- 首次出现的术语就地给一句大白话：CRO=不花广告费靠优化让流量多成交；pp=百分点；SoV/SoP=曝光/成交份额；CTR/CVR=点击率/转化率；ACOS=广告花费÷广告销售额。
- 每次委派"审查"类子代理，除业务/UX 视角外，**必带一个"非运营外行能否看懂"的视角**；外行看不懂的，负责翻译。
- 聊天回复同样避免堆术语，先给大白话结论再展开。


## 命名纪律

- 作战表迭代叫"初版/修订版/定版"；**"V2/V3" 只指仪表盘**（别用重）。
- 文件名英文外壳、内容中文；难排序的用零填充数字前缀（ADR 的 `0001-`）。


## 多子代理审查

用户偏好用子代理做**对抗性审查/正反辩论**来验证结论（业务视角 + UX 视角 + 外行视角）。重要结论落地前先派子代理证伪，再自己复算把关。本会话多次靠此抓出真错（作战表方法、Q1 CVR 诊断）。
