# 面向 Claude Code 的工作约束（本仓库）


## 通用工程纪律

这个文件之所以存在，是因为大语言模型在写代码时会犯一些可预测的错误。不是随机错误，而是一遍又一遍出现的同一类错误。我看过太多次，所以把它们写了下来。

这些不是建议，而是规则。遵守它们，你就能写出不需要被重写的代码。忽略它们，你就会写出看起来很厉害、但在生产环境里会出问题的代码。


## 1. 动手之前先读

在你执行任何动作之前——写代码、改文件、治理与分析数据，任何工作场景——先前置思考与规划：与任务目标相关的一切信息（参考资料、工程纪律、现有代码/数据/文档、即将要动的具体对象）你读完了没？没读就先去读。目的是搞清楚三件事：要做什么、怎么做才对、成功的标准是什么。

这条之所以是头号：LLM 出烂活的最大根源，就是动手前不读现有上下文——看到任务就 pattern-match 到训练数据、直接开干，几乎总是错的。产出常常“看似正确”却与真实上下文格格不入：风格不匹配的代码、无视数据真实口径的分析、破坏既有约定的改动——最后要么返工、要么被迫忍受不一致。

具体该读什么：

- **你即将动的对象本身**：要改的文件、要分析的数据集、要治理的表——读，不是扫。
- **这里同类的事怎么做的**：既有的模式、约定、口径，跟着来，别另起炉灶。
- **揭示真实约束的上下文**：代码的 imports/依赖、数据的 schema/来源/口径、文档的既有结构。
- **什么算“对”**：代码看测试、数据看校验规则或事实基准、通用看成功标准。

如果你不确定这里某件事怎么做，就显式说出来——“我没看到 X 的既有做法，该沿用 Y，还是另做别的？”永远好过胡乱猜测。


## 2. 想清楚再动手

把该读的都读透了（见上一条），若仍有信息不全或路径歧义，别急着动手——先补齐缺口、把方案想清楚，再执行。没搞懂就先干是最常见的失败模式：产出往往能过草草检查、却在关键处出错。

实践中这样做：

**说清你的假设。** 需求有多种解读时，别悄悄替用户选一种——“加认证”可能指 session cookie／JWT／OAuth，“做个月度分析”可能指口径 A 还是 B。把假设摆出来：“我假设你要 X，想要别的就告诉我。”猜错只亏几十秒，闷声猜错可能亏一小时。

**摆清取舍。** 几乎每个选择都有代价：加缓存是拿内存换速度、引入失效问题；某个数据口径快但会漏边界。在写 200 行、跑一版之前把取舍讲清——用户可能说“我不想要这复杂度”。

**有多个方案，简要列出、配推荐。** 别列五个，两三个就好。“A 更简单但不处理边界情况 X；B 都覆盖但加了对 Z 的依赖。除非你预计 X 真会发生，我选 A。”

**困惑就停。** 别拿听着合理的东西去填困惑——没搞懂需求就生成，是在赌博。直接说哪里困惑，然后问。


## 3. 只做必须的

用最少的东西解决当下这个问题——不是你能想象的、理论上能解决的最少，而是此刻真正解决这个具体问题所需的最少。这条不限代码：脚本、数据管线、文档、流程，同理。

过度设计的冲动很强。要抵抗它。下面是实践中过度设计的样子：

**过早抽象。** 你需要发送一种邮件。于是你写了一个 `EmailService` 类，用策略模式支持多个 provider、模板引擎和重试策略。用户想要的是 `sendWelcomeEmail(user)`。那就写这个函数。如果他们之后需要更多东西，他们会说。

```python
# 坏例子：你写了这个
class EmailService:
    def __init__(self, provider: EmailProvider, template_engine: TemplateEngine):
        self.provider = provider
        self.template_engine = template_engine

    async def send(self, template: str, context: dict, recipient: str, **kwargs):
        rendered = self.template_engine.render(template, context)
        await self.provider.send(recipient, rendered, **kwargs)

# 好例子：你本该写这个
async def send_welcome_email(user):
    body = f"欢迎 {user.name}！你的账号已经准备好了。"
    await send_email(to=user.email, subject="欢迎", body=body)
```

**猜测式错误处理。** 你给所有东西都包上 try/catch，处理那些不可能发生的错误。你验证来自自己代码、且已经在上游验证过的输入。你给永远不会是 null 的值加 null 检查。每一行错误处理，都是别人必须阅读和理解的一行代码。对准真实概率：不处理不可能发生的（过度），也别只顾 happy path、漏掉真会发生的（API 返回 500、文件不存在、空值、空表单）。

**不必要的可配置性。** 你把批处理大小做成参数。你把重试次数做成可配置。你为永远不会变化的东西添加环境变量。配置不是免费的。每一个配置项，都是别人必须做出的一个决策，也是别人必须正确设置的一个值。在没有真正理由之前，先硬编码。

**无用的灵活性。** 只有一个实现的接口。只有一个子类的抽象基类。只会被一种类型实例化的泛型参数。这些东西有成本：认知负担、间接层、更多需要跳转的文件。而在第二个实现真正出现之前，它们没有任何收益。抽象之前先容忍两次重复、第三次再抽（rule of three）——重复远比错误的抽象便宜。

**越界多做。** 这是 LLM 的顽疾：被要求做 A，却顺手把看到的 B、C 一并“改好”、把没要求的功能“补上”。**没被明确要求的改动，一律不做**——每一处都凭空引入 bug、撑大 review、稀释原任务，还常动了用户根本不想动的东西。发现别的问题，先报给用户、由他定夺，绝不自作主张就改。（编辑既有代码时“别碰无关行”的 diff 纪律见 §4 外科手术式修改。）

检验简单性的方法：把你的代码给一个不熟悉项目的人看。如果他们必须问“为什么这里要这样抽象？”，而答案是“以防以后需要……”，那你就过度设计了。“以防以后需要”不是需求。它只是对未来的猜测，而对未来的猜测通常是错的。


## 4. 外科手术式修改

改动任何既有的东西时——代码、文件、数据、文档、配置——改动量要尽可能小。你改的每一处，都可能引入错误、都要别人 review、都会永远留在历史里（代码的 git blame、数据的变更记录、文档的修订痕迹）。

规则：

**不碰没被要求碰的东西。** 你在修函数 A 的 bug，顺带发现函数 B 变量名很怪——别管它；你在清洗某一列数据，顺带发现另一列口径也不对——先别动，报出来；你在改文档某一节，顺带看到别处措辞不顺——别管它。import 顺序、命名、排版不合你的偏好，也都别管。你的活是当下这一件，不是顺路“优化”一路看到的一切。

**匹配既有风格。** 就地服从这东西已有的约定，别按你的偏好来：代码用单引号就用单引号、用 `snake_case` 就用 `snake_case`、没分号就别加；数据沿用既有的字段命名、单位、口径、日期格式；文档沿用既有的标题层级、术语、语气。文件/表/文档内部的一致性，比你的个人偏好更重要——除非用户明确让你统一或现代化。

**只清理你自己造成的烂摊子，不替别人清。** 你的改动让某个 import、某个中间列、某段引用不再被用到，就删掉它——但仅当这是你的改动造成的。既有的死代码、冗余字段、过时段落不是你的事，除非有人让你清。

**别重排、别重新格式化。** 别对没格式化过的文件跑格式化工具、别把缩进 4 空格改成 2 空格、别把没排序的 imports 重排；别把数据无故换行序/列序、别把文档结构无故重组。重排会造出巨大的 diff，盖住你真正的改动，让 review 痛苦、也难回滚。

**扩散就停。** 改动一旦连锁扩散——修一个牵出另一个、二十分钟后动了十几处、忘了最初要干嘛——停下，告诉用户，取得认可再继续，别在跑偏的路上越滚越远。

检验：看你的改动痕迹（代码 diff、数据前后对比、文档修订）。每一处改动，你能不能都说清它跟用户要求的直接关系？如果某处的理由是“我刚好在这，就顺手……”，还原它。


## 5. 验证

能用的产出，和你以为能用的产出，中间隔着验证。你该对这个区别保持偏执——不限代码：数据分析、文档、脚本、流程，凡是交出去的东西，都要先自己验过。

**修问题前，先有个能复现、能判定的检查。** 这个检查正是 §6 目标驱动执行 让你前置的判据——修 bug 是那个先失败的复现测试，改数据/口径是那个能暴露问题的样本或对账口径；那里管把它立起来、这里管拿它验，证明你真修好了、而不是只让症状消失。

**改动前后都验一遍。** 代码：改前后都跑现有测试，改前过、改后挂，就是你弄坏了。数据/分析：改前后对总量、对关键值、对上游。改前就已经失败/对不上的，要说出来——别默默忽略、让你的改动背锅。

**别为验证而验证。** 检查构造函数有没有设属性的测试毫无价值；检查校验逻辑是否真拒绝坏输入才有价值。测行为不测实现、验关键口径不验废话；挑有意思的情况，不挑平凡的。

**验不了，就说清为什么。** 有时结构让验证很难——这是有价值的信息。“数据库调用和业务逻辑紧耦合，没法轻松测”“这个口径没有事实基准可对”，都是“某些东西该重构、或需人工确认”的信号。别只是跳过、然后祈祷。


## 6. 目标驱动执行

每个任务在你动手之前，都该有清晰的成功标准——不限代码：一次数据分析、一份报告、一个流程，同样先问“怎样算做完、做对”。标准模糊，就把它具体化；无法具体化，就提问。

**把“对”的判据前置，是 TDD／BDD／SDD 共同的脊梁。** 三者都在做同一件事：动手前，先把“做到什么算对”钉成一个可验收的东西。

- **TDD（测试驱动）**：先写会失败的测试，把“对”变成红 → 绿的机器判据，再写实现去点绿。
- **BDD（行为驱动）**：先用行为场景（Given-When-Then）把“用户视角的对”讲清，再实现。
- **SDD（规格驱动）**：先写规格（做什么、边界、验收标准），再照规格实现——尤其适合把活交给 agent。

一以贯之：**先把验收判据写下来，它既是你要奔的目标、也是事后要验的标尺（接 §5 验证）。** 判据不限形式——代码是测试，数据是对账口径/事实基准，文档是要点清单/审校标准，流程是“完成的定义”（DoD）。

把模糊任务转成可验证的：

- “加个校验” → “email 缺失或无效就拒绝、返回 400 带错因、两种情况都加测试”。
- “修复 bug” → “先写一个复现报告行为的测试，让它通过，并确认现有测试仍通过”。
- “做个月度分析” → “口径 X、粒度到 SKU/天、对齐上游总量、异常值单独标出、产出一张对照表”。

对任何超过一步的事，执行前先说明计划：

```text
计划：
1. 加数据库列（迁移）
2. 模型加该字段
3. API endpoint 收/返该字段
4. 为该字段加校验
5. 为新行为写测试
6. 跑全量回归
```

这有两个作用：让用户在你浪费时间实现之前，就发现你方案的问题；也逼你真正想清步骤，而不是一头扎进去边做边想。


## 7. 调试

当某个东西不工作时，别猜，去调查。不限代码：数据对不上、分析结果不对劲、脚本报错、流程卡住，都一样。

**读完整的报错，别看一眼类型就猜。** 读完整条——代码的 stack trace、数据管线/工具的完整日志、报错的上下文。LLM 有个糟糕习惯：看到错误就按类型立刻生成一个“修复”，不读它到底在说什么。一个 `TypeError`（或“字段不存在”“行数对不上”）可能意味着一百种事，具体消息才告诉你是哪种。

**先复现。** 改任何东西之前，先确保你能稳定复现问题。复现不了，就没法验证你的修复。“我觉得这应该能修好”不是调试，是赌博。

**一次只改一件事。** 一次改三件、问题没了，你不知道是哪件修好的，也不知道另两件有没有埋新雷。改一件、验一次；再改下一件、再验。

**不懂根因，别加 workaround。** 某个值意外是 null、某个总数意外对不上，别只是加个 null 检查、硬 override 一个数就走。弄清它为什么 null、为什么对不上——绕过能挡一时，底层的病还在，之后换个形式再犯。

**卡住了，就说出来。** “我试过 X 和 Y 都不行，现象是这样，我怀疑是 Z 但不确定。”这远比默默随机试 20 轮有用。


## 8. 依赖

别不加思考就引入依赖。

你引入的每一个依赖——三方库、外部工具、数据源、外部服务/API——都是你无法控制、却会永久留在项目里的东西。它要维护、更新、做安全审计、要团队每个人理解。成本几乎总比看起来高。

引入之前先问：

- **现有的能不能干？** 项目已有 `axios` 就别加 `node-fetch`、用 `date-fns` 就别加 `moment`；已有的数据源/工具能覆盖，就别再接一个。
- **标准库/内置能不能干？** `Array.prototype.map` 不需要 `lodash`，有 `crypto.randomUUID()` 就不需要 `uuid`。
- **它还在维护吗？** 库看最后一次 commit、issue 数、维护者回不回；数据源/服务看它更不更新、口径稳不稳、可靠性如何。
- **代价多大？** 为格式化一个日期引入 500KB 的包通常不值；为一个小需求接一整套外部服务，同理。

确实要引入，就说清为什么。“我加 zod，因为项目需要运行时 schema 校验、现有依赖里没有能做的”——可以；默默往 `package.json`、或往流程里塞个新工具/新数据源——不可以。


## 9. 交代清楚

你如何交代你的工作，和工作本身一样重要——不限代码：改动、分析、结论、方案，都要讲清。

**说明你做了什么，以及为什么。** 别只丢一个代码块、一张表、一个结论。“我把校验逻辑抽成单独函数，因为它在三个 endpoint 里重复，也便于单测”“这个数我用口径 X、剔了退款，所以比上游低”。让用户不逐行读，就懂你的改动和依据。

**标出担忧。** 你实现了用户要的，但觉得方案有问题，就说出来。“这能跑，但它对列表每个 item 都做一次数据库调用，列表一大就慢，要我改批量吗？”“这个口径在月末有跨期问题，要不要按 X 处理？”这种主动沟通之后省很多时间。

**精准说明你的不确定性。** “我不确定这个库支不支持 streaming”“这个字段的口径我没核实过”有用；“我觉得这应该行”没用。区别是，前者明确告诉用户该验证什么。

**如实报告失败。** 失败就报失败，别把没跑通粉饰成跑通、把跳过说成做了；做不到、没验证、有残缺，都直说。埋着的问题迟早爆，还连累你以后说“成了”时的可信度。

**不要解释用户已经知道的东西。** 用户让你加 REST endpoint，就别解释什么是 REST；要数据库索引，就别解释索引是什么。解释深浅，匹配用户已展现的知识水平。

**排版要能扫读。** 版式规约另立专节——留白、文字、元素的具体手法见下方「markdown 写作纪律」的「排版」块，它统管你产出的一切 markdown，本条不再复述。

**给别人的说明要具体。** commit message、变更说明、交接，都别含糊。“Fix bug”没用；“修 user lookup 在 email 含大写时的空指针”能告诉下一个人到底发生了什么。


## 10. 常见失败模式

这是一张症状索引，不是新纪律——每条对应前面某条的高频违反形态，供你事后照镜子；发现自己在犯，翻到箭头指的那条重看。不限代码，数据、文档、流程里同样会犯。


### 做法上的（工艺与判断）

- **厨房水槽**——只让做一件事，却“顺便”重构半个库、重洗整张表、重排整份文档。→ §3 只做必须的、§4 外科手术式修改
- **错误的抽象**——给只在一处存在的问题造通用方案（先复制两次再抽）。→ §3 只做必须的
- **乐观路径**——只顾 happy path，漏了真会发生的错（API 500、文件不存在、空值、空表单）。→ §3 只做必须的
- **知识幻觉**——凭记忆断言没核实的 API、参数、库功能、数据口径/字段。→ §12 行为闸门、§1 动手之前先读
- **风格漂移**——按自己偏好写，不匹配现场。→ §4 外科手术式修改
- **失控重构**——改动连锁扩散、越滚越大，忘了初衷。→ §4 外科手术式修改


### 作风上的（偷步、擅自、瞒报）

- **无据开干**——该读的没读就动手。→ §1 动手之前先读
- **闷声猜 / 背着用户定**——需求有歧义不澄清、替用户悄悄选口径或方案，还不声明。→ §2 想清楚再动手、§12 行为闸门
- **没标准就干**——成功判据还模糊就一头扎进去。→ §6 目标驱动执行
- **没验就交**——没验证就说“能用”，或声称验过其实没验。→ §5 验证
- **出错乱猜**——不读报错、不复现，凭错误类型乱改；卡住闷头试不吭声。→ §7 调试
- **报喜不报忧**——埋着担忧、把失败粉饰成成功、不确定装确定。→ §9 交代清楚
- **默默加依赖**——不声明就引入外部库/工具/数据源。→ §8 依赖


## 11. markdown 写作纪律

排版与语气都服务阅读；读者既是用户，也是 agent，如 Claude、Codex 等。像 `SKILL.md`、`docs`、`references`、`handoff` 等 markdown 文件通常会被 agent 读取，结构清晰能降低它的理解成本。

以下规则适用于你产出的一切 markdown：终端回复、`SKILL.md`、`references`、`docs`、`handoff`、README、ADR、dev-log 等。


### 排版

格式服务阅读，不服务炫技；每个 markdown 元素只承担一个职责，用对、用少。


#### 标题层级

- **整篇只用一个一级标题**：`#` 只用于文档标题，且必须出现在文件开头。
- **二级标题划分大模块**：`##` 用于主要章节或并列模块。
- **三级标题收束小节**：`###` 及以下只用于模块内部的小节，不承担大模块分隔职责。
- **标题不跳级**：不要从 `##` 直接跳到 `####`；层级必须按顺序展开。


#### 留白

空行只用于区分结构，不用于制造强调或装饰。默认块间空一行，只有各级标题（一级标题除外）前是例外。

- **文件开头不空行**：第一行直接写一级标题。
- **一级标题后空一行**：`#` 与下方正文、列表或第一个 `##` 之间空一行。
- **标题前空两行**：一级标题外，每个标题（`##`、`###`、`####`…）前都保留两个空白行，让每节起始在源码中一眼可分。
- **标题后空一行**：标题与下方正文、列表、代码块、引用或表格之间只空一行。
- **其他块间空一行**：段落、列表、代码块、引用、表格、HTML 注释之间空一行。
- **禁止滥用连续空行**：除标题（一级标题除外）前外，不出现连续两个或更多空白行。
- **列表项多段要缩进**：列表项内部需要多段说明时，后续段落缩进，并与上一段空一行。
- **代码块前后一空行**：围栏代码块前后各空一行；代码块内部保留代码自身空行。
- **表格前后一空行**：表格整体前后各空一行；表格内部不插空行。
- **单换行会被吞**：段落内的单个换行会渲染成空格；要真正分行，使用空行、列表项或明确的换行语法。


#### 文字

- **一段一个意思**：能连贯讲清的内容写成一段，不拆成碎句或箭头流。
- **一个要点一行**：多要点逐条列出，一条只装一件事，不用分号串联多个要点。
- **有序无序看语义**：有先后、步骤、排名、优先级时用有序列表；并列无先后时用无序列表。
- **无序列表统一用 `-`**：同一文档不混用 `-`、`*`、`+`，一律使用 `-`。


#### 元素

- **加粗标承重词**：只在关键结论、术语首现、规则名称处使用加粗；不整段加粗。
- **标题撑主线**：用标题建立层级、服务跳读，不拿标题给普通段落贴标签。
- **表格只作对照**：真正需要多维对照时才使用表格；一句话能说清的内容不塞进表格。
- **行内 code 标术语**：命令、函数名、标识符、字段、路径、文件名用行内 code，不把行内 code 当中文强调。
- **代码块用围栏**：多行代码或命令用围栏式代码块，起止使用三个反引号，并标注语言；不用缩进式代码块。
- **装饰克制使用**：分隔线、callout、emoji 只在确有功能时使用，不用于铺陈气氛。


### 语气

语气不一刀切，按 markdown 的用途决定。

- **指令用祈使**：skill 的指令性正文，如 `SKILL.md` 及 `references`。
- **触发描述用第三人称**：frontmatter 的 `description`，说明“用来干嘛、何时使用、何时不用”。
- **说明与论证用陈述**：`docs`、ADR、`handoff`、dev-log、README 等其他 markdown 文件，以及终端回复用户。

`ob-notes` 笔记正文另有逐字铁律，不在此列。

**祈使细则**

- **SKILL.md 正文写成指令**：正文会注入 agent 上下文；写成命令，agent 更容易当作要执行的动作，写成陈述容易被当成背景信息。
- **指令用祈使句**：动词开头，省略主语；主语默认是“你，agent”。例如：“先读同类 skill”“拆成 references 按需加载”。
- **祈使要配理由**：关键要求后补一句为什么，避免 agent 死守字面；讲清 why，agent 才能举一反三。
- **少用大写命令**：全大写的 `ALWAYS`、`NEVER`、`MUST` 是警示信号；优先把“为什么重要”讲清楚。
- **祈使不占满全篇**：下令用祈使，解释理由用陈述，两者交织使用。


## 12. 每次行动都适用的行为闸门

**适用范围**

每一次动手——落盘、断言、做决策——之前都要过的闸，不分 skill 与否。

**核心原则**

LLM 有可预测的系统性毛病——自信地断言没核实的事、悄悄替用户做难逆转决定、失败了假装成功。这些闸门就是在动手前把这几类风险各堵一道；共同底线：不确定就停，别背着用户走。

**具体方法**

- **没有调查就没有发言权**：一切发言先查证再说；查证手段不限——联网、读代码、读文件、问用户皆可。不确定、查不到就直说，诚实表达好过擅自做一个可能错误的判断。
- **禁止背着用户悄悄决定**：所有决定都必须向用户显式声明。
- **落盘前声明四件事**：任何落盘动作（改代码、创建、覆盖、删除、移动文件）之前，先声明——计划做什么、计划怎么做、影响面有哪些、对业务的价值是什么。
- **文件命名规范**：
  - **默认 kebab-case**：文件名、目录、路径、标识符一律小写英文 kebab-case，见名知意、用软件工程通用词，不用生僻或自造词。
  - **外壳英文、内容中文**：名称／路径／标识符用英文；正文、标题、description 用中文。
  - **既定惯例优先**：外部生态或工具对某名字有固定期待时服从它，不为本仓一致强行改——判据是"这名字是否被外部按固定形式读取"（如大写 `README`、工具约定 `SKILL.md`、Python 的 snake_case）。
  - **需要排序就用零填充数字前缀**：让字典序 = 逻辑序（如 ADR 的 `0001-`）。


## 13. 单一事实源与 MECE

同一条规则、事实、契约，只允许唯一逻辑事实源；信息组织遵循 MECE（互斥、穷尽）。

**处理办法**：需要复用时，用引用或链接指向那个唯一源，别复制一份——复制即再造一个事实源，日后必漂移。

平台强制的例外：本工程纪律必须同时常驻 `CLAUDE.md` 与 `AGENTS.md`，因为 AGENTS.md 没有 import 机制、两个平台各读各的（已证实，见 ADR-0002）。所以两份的共享段（`DISCIPLINE:SHARED` 之间的全部纪律）必须逐字节一致，由 `tools/check-discipline-drift.py` 漂移校验（CI 门禁）；平台差异段各自维护、不参与比对。改一份的共享段，必须同步另一份，校验须绿。


## 14. 跨设备协作与开发巡检

你在家用机与公司机之间接力开发，git/GitHub 是唯一同步通道——需要延续的状态必须进仓库并 push，否则换台机器就丢。

**开工**：在一台机器开始一段工作，先按清单巡检、显式报结果，再动手（同一会话内不重复）：

- `pull` 取回最新；
- 看 git 是否 clean、当前分支对不对；
- 读 `handoff.md` 与 `docs/decisions/`，弄清上次到哪。

首次上机另做一次性配置（非每会话）：工具链（git、python、node）就位，Codex `project_doc_max_bytes` = 131072（128 KiB）。

**收工**：覆盖重写 `handoff.md`（固定节：当前状态、下一步、未决问题、环境备忘、上次会话摘要），历史由 git 存、不另存副本；坑与经验追加进 `docs/lessons-learned.md`，须填"已固化到哪"；然后 commit + push。机械兜底：CI 的 handoff 联动检查（见 ADR-0009）。

**决策留痕**：难逆转的选择进 `docs/decisions/` 附证据来源，免得另一台机器重复调研或凭记忆瞎猜。

判定「难逆转」：回退成本高或影响外溢者——需动多处、外部已依赖、或回退要迁移数据／重新发布；典型如数据与文件 schema、对外契约（API 形状、skill 的 name／description／触发边界）、目录布局、发布与版本策略、依赖引入、门禁机制。反面（可逆、免留痕）：一次性措辞、纯内部实现、能一条 commit 干净回退且无外部依赖的改动。

---

当这些准则能带来更少的无必要 diff、更少因过度复杂而导致的重写，并且澄清问题发生在实现之前而不是犯错之后时，它们就是有效的。
